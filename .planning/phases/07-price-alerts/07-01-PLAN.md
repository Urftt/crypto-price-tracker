# Plan 07-01: Backend — PriceAlert Model, SQLite CRUD, Alert Checking Logic, and Unit Tests

**Phase:** 7 — Price Alerts
**Depends on:** Phase 6 (uses same DB file and identical patterns from `portfolio_db.py`)
**Estimated tasks:** 3

## Goal

Build the complete backend for price alerts: the `PriceAlert` dataclass, SQLite storage layer with CRUD operations (add, get active, get all, remove, mark triggered, clear triggered), a pure alert-checking function, and comprehensive tests for all of it. This plan produces the foundation that Plan 07-02 wires into the CLI and web surfaces.

## Context

**Existing patterns to follow:**
- `@dataclass(slots=True)` in `models.py` (see `CoinData`, `Holding`)
- `portfolio_db.py` connection-per-call pattern: `get_connection()` with WAL mode, row factory, `_ensure_schema`, try/finally close
- `portfolio_db._get_default_db_path()` resolves `~/.local/share/crypto-tracker/portfolio.db` — alerts table lives in the **same** DB file
- Tests use pytest with `uv run pytest tests/ -x -q`

**Key architectural decisions (from RESEARCH.md & CONTEXT.md):**
- `alerts_db.py` mirrors `portfolio_db.py` exactly — own `_alert_factory`, own `_ensure_alert_schema`, same DB file
- **Import pattern for shared DB path:** `alerts_db.py` uses `from crypto_price_tracker import portfolio_db` (module-level import) then calls `portfolio_db._get_default_db_path()` through the module reference. This ensures test fixtures that patch `portfolio_db._get_default_db_path` also affect `alerts_db` calls.
- Pure `check_alerts(coins, active_alerts) -> list[PriceAlert]` function — no DB side effects in the checking logic
- Race condition guard: `UPDATE alerts SET status='triggered' ... WHERE id=? AND status='active'`
- Direction validation via SQLite CHECK constraint: `CHECK(direction IN ('above', 'below'))`
- Status validation via SQLite CHECK constraint: `CHECK(status IN ('active', 'triggered'))`
- `datetime.now().isoformat()` for timestamps (ISO 8601 consistency)

**Step 2: Create alerts_db.py**

This module handles all SQLite interactions for the alerts table. It mirrors `portfolio_db.py` exactly but uses its own row factory and schema function. Both tables live in the same `portfolio.db` file. Every public function accepts an optional `db_path: Path | None` parameter so tests can pass a temp file path.

**Imports:**
```python
from __future__ import annotations

import sqlite3
from datetime import datetime
from pathlib import Path

from crypto_price_tracker.models import PriceAlert
from crypto_price_tracker import portfolio_db
```

**CRITICAL: Import `portfolio_db` as a module, NOT `from portfolio_db import _get_default_db_path`.** This is essential so that test fixtures can patch `portfolio_db._get_default_db_path` and have the change visible to `alerts_db` calls. If we used `from ... import _get_default_db_path`, the local binding would capture the original function at import time and patching the module attribute wouldn't work.

**Functions to implement (in this order):**

1. `_alert_factory(cursor: sqlite3.Cursor, row: tuple) -> PriceAlert` — Custom row factory: extract column names from `cursor.description`, zip with row values, construct `PriceAlert(**dict(zip(names, row)))`. Same pattern as `_holding_factory` in `portfolio_db.py`.

2. `_ensure_alert_schema(conn: sqlite3.Connection) -> None` — Execute:
   ```sql
   CREATE TABLE IF NOT EXISTS alerts (
       id            INTEGER PRIMARY KEY AUTOINCREMENT,
       symbol        TEXT NOT NULL,
       target_price  REAL NOT NULL CHECK(target_price > 0),
       direction     TEXT NOT NULL CHECK(direction IN ('above', 'below')),
       status        TEXT NOT NULL DEFAULT 'active' CHECK(status IN ('active', 'triggered')),
       created_at    TEXT NOT NULL,
       triggered_at  TEXT
   )
   ```

3. `get_alert_connection(db_path: Path | None = None) -> sqlite3.Connection` — Open connection to `db_path or portfolio_db._get_default_db_path()`. Set `PRAGMA journal_mode=WAL`, set `PRAGMA foreign_keys=ON`, assign `conn.row_factory = _alert_factory`, call `_ensure_alert_schema(conn)`, return conn. Same pattern as `get_connection()` in `portfolio_db.py` but with its own row factory. **Note:** Call through `portfolio_db._get_default_db_path()` (module reference), NOT a local import.

4. `add_alert(symbol: str, target_price: float, direction: str = "above", *, db_path: Path | None = None) -> int` — Validate direction is "above" or "below" (raise `ValueError` if not). Upper-case symbol. Set `created_at = datetime.now().isoformat()`. Open connection via `get_alert_connection(db_path)`. `INSERT INTO alerts (symbol, target_price, direction, status, created_at) VALUES (?, ?, ?, 'active', ?)` inside `with conn:`. Return `cursor.lastrowid`. Close conn in `finally` block.

5. `get_active_alerts(*, db_path: Path | None = None) -> list[PriceAlert]` — `SELECT * FROM alerts WHERE status = 'active' ORDER BY created_at`. Return `cursor.fetchall()`. Close conn.

6. `get_all_alerts(*, db_path: Path | None = None) -> list[PriceAlert]` — `SELECT * FROM alerts ORDER BY CASE WHEN status = 'active' THEN 0 ELSE 1 END, created_at`. This returns active alerts first, then triggered alerts (matching the `crypto alert list` display requirement). Return `cursor.fetchall()`. Close conn.

7. `remove_alert(alert_id: int, *, db_path: Path | None = None) -> bool` — `DELETE FROM alerts WHERE id = ?`. Return `cursor.rowcount > 0`. Close conn.

8. `mark_triggered(alert_id: int, *, db_path: Path | None = None) -> bool` — `UPDATE alerts SET status = 'triggered', triggered_at = ? WHERE id = ? AND status = 'active'` with `datetime.now().isoformat()` as the timestamp. The `AND status = 'active'` guard prevents double-triggering in concurrent requests. Return `cursor.rowcount > 0`. Close conn.

9. `clear_triggered_alerts(*, db_path: Path | None = None) -> int` — `DELETE FROM alerts WHERE status = 'triggered'`. Return `cursor.rowcount` (number of deleted rows). Close conn.

**Important:** Every function that opens a connection must close it after use. Use `try/finally` to guarantee cleanup, same pattern as `portfolio_db.py`.

**Verify:** `uv run python -c "from crypto_price_tracker.models import PriceAlert; from crypto_price_tracker.alerts_db import get_alert_connection; import tempfile, pathlib; p = pathlib.Path(tempfile.mktemp(suffix='.db')); c = get_alert_connection(p); print('OK:', p); c.close()"`

### Task 2: Create `alerts.py` — Alert checking service
**File:** `src/crypto_price_tracker/alerts.py`
**Action:** create

This module contains the pure alert-checking function. It does NOT touch the database — it takes a coins list and active alerts list, and returns which alerts triggered. The caller is responsible for calling `mark_triggered()` on each returned alert.

**Imports:**
```python
from __future__ import annotations

from crypto_price_tracker.models import CoinData, PriceAlert
```

**Function to implement:**

`check_alerts(coins: list[CoinData], active_alerts: list[PriceAlert]) -> list[PriceAlert]` —
- Build a price lookup dict: `prices = {c.symbol: c.price for c in coins}`
- Iterate over `active_alerts`. For each alert:
  - Get `current_price = prices.get(alert.symbol)`. If `None` (coin not in fetched data), skip — the alert stays active but doesn't trigger.
  - If `alert.direction == "above"` and `current_price >= alert.target_price`: add to triggered list.
  - If `alert.direction == "below"` and `current_price <= alert.target_price`: add to triggered list.
- Return the triggered list.

**Why pure:** Keeping the checking logic side-effect-free makes it trivially testable without DB fixtures. The caller (CLI `cmd_prices`, `cmd_watch`, `cmd_alert_check`, or web `/api/prices`) handles the DB update.

**Verify:** `uv run python -c "from crypto_price_tracker.alerts import check_alerts; print('import OK')"`
